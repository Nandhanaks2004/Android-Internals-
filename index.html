Linux Fundamentals

Android is built on the Linux kernel. Understanding Linux fundamentals is essential for Android security analysis and penetration testing.

2.1.1 — Linux Kernel in Android

The Linux kernel is the heart of Android. It handles hardware communication, memory management, process scheduling, and security enforcement.

🧠

Kernel Responsibilities

  • Process scheduling (CFS — Completely Fair Scheduler)
  • Memory management (virtual memory, paging, OOM killer)
  • Hardware abstraction (drivers for camera, WiFi, GPS, etc.)
  • Security enforcement (SELinux, DAC, capabilities)
  • Inter-process communication (IPC)
  • Power management and wake locks
📱

Android Kernel Modifications

  • Binder IPC — custom IPC replacing traditional pipes
  • Wakelocks — prevent CPU sleep for background tasks
  • Low Memory Killer (LMK) — aggressive process killing
  • Ashmem — anonymous shared memory for efficient IPC
  • SELinux (enforcing) — mandatory access control
  • Seccomp — system call filtering
Android Architecture Stack
Android Apps
↓
Java/Kotlin API
NDK (Native C/C++)
↓
Android Framework
↓
Hardware Abstraction Layer (HAL)
↓
Linux Kernel
⚡ Pentester's Perspective

The Android kernel is a prime target. A kernel exploit gives full system access (root), bypassing all sandboxing:

  • Kernel vulnerabilities — CVE databases for unpatched exploits (DirtyPipe, DirtyCow)
  • Driver vulnerabilities — Qualcomm, MediaTek, Samsung drivers are common targets
  • SELinux bypass — Misconfigured policies can be exploited
  • Bootloader exploitation — If unlocked, allows custom kernel loading
🎯 Key Concept
The Linux kernel version can be found via uname -r or in /proc/version. Always check kernel version to determine which exploits may work during a penetration test.

2.1.2 — Shell Basics

The shell is your command-line interface to the Android system. Through ADB shell, you interact directly with the Linux subsystem.

# Print current directory
pwd                          # /data/data

# List directory contents
ls                           # basic listing
ls -la                       # detailed + hidden files

# Change directory
cd /system                   # go to /system
cd ..                        # go up one level
cd ~                         # go to home directory
# Create files and directories
touch file.txt              # create empty file
mkdir -p dir/subdir          # create nested dirs

# Copy, move, delete
cp source.txt dest.txt     # copy file
mv old.txt new.txt         # move/rename
rm file.txt                 # delete file
rm -rf dir/                  # force delete dir
# View file contents
cat file.txt                # print entire file
head -n 20 file.txt        # first 20 lines
tail -n 20 file.txt        # last 20 lines

# Filter and process
grep -r "password" .       # recursive search
sort file.txt               # sort lines
wc -l file.txt              # count lines

# Pipe and redirect
cat log.txt | grep "error" | wc -l
echo "hello" > out.txt     # overwrite
echo "world" >> out.txt    # append
# Find files by name
find / -name "*.apk" 2>/dev/null

# Find files by size
find /data -size +10M -type f

# Find recently modified files
find /sdcard -mtime -1 -type f

# Find SUID binaries (privesc)
find / -perm -4000 -type f 2>/dev/null

2.1.3 — Process Management

Android runs dozens of processes simultaneously. Understanding process management helps identify running services and detect malware.

⚙️

Process States

  • Running (R) — Actively executing on CPU
  • Sleeping (S) — Waiting for an event
  • Stopped (T) — Suspended by signal
  • Zombie (Z) — Terminated but not reaped
📋

Key Process Commands

  • ps -A — List all processes
  • ps -ef — Full format listing
  • top — Real-time process monitor
  • kill -9 PID — Force kill process
🔍 Simulation: Process Lifecycle in Android
1
App Launch: User taps icon → Zygote forks new process → App process starts
2
Running: App in foreground, CPU allocated by CFS scheduler
3
Background: User presses Home → App moves to background, lower CPU priority
4
Memory Pressure: LMK targets background processes with highest oom_adj score
5
Termination: LMK sends SIGKILL → Process terminated → Memory freed
6
Restart: If app has foreground/sticky service, system may restart it
🔐 Section 2.1.4

File Permissions (chmod/chown)

Linux file permissions control who can read, write, and execute files. On Android, these work alongside the Android permission system for layered security.

🔐

Permission Types

  • r (read) — View file contents / list directory
  • w (write) — Modify file / create-delete in dir
  • x (execute) — Run as program / access directory
  • s (setuid) — Run as file owner (dangerous!)
  • t (sticky bit) — Only owner can delete file
👤

Ownership Classes

  • u (user/owner) — The file's owner
  • g (group) — Users in the file's group
  • o (others) — Everyone else
  • a (all) — All three classes (u+g+o)
⚠️ Permission Number System

r=4, w=2, x=1. Combine for each class:

chmod 755 file → Owner=rwx(7), Group=r-x(5), Others=r-x(5)

chmod 644 file → Owner=rw-(6), Group=r--(4), Others=r--(4)

chmod 700 dir → Owner=rwx(7), Group=---(0), Others=---(0)

# View permissions
ls -la /data/data/com.app/
# drwx------ 6 u0_a45 u0_a45 4096 databases

# Change permissions (symbolic)
chmod u+rwx file.txt       # add rwx for owner
chmod g-w file.txt         # remove write for group

# Change permissions (numeric)
chmod 755 script.sh        # rwxr-xr-x
chmod 600 secret.key       # rw-------

# Change ownership
chown u0_a45:u0_a45 file
chown -R root:root /system

# Find SUID/SGID files (privesc vectors)
find / -perm -4000 -type f 2>/dev/null  # SUID
find / -perm -2000 -type f 2>/dev/null  # SGID
⚡ Pentester's Perspective
  • World-readable app data — Misconfigured permissions expose sensitive data
  • SUID binaries — Run as root, common privilege escalation vectors
  • Weak /data permissions — Apps can read each other's data
  • SELinux context matters — Even with correct DAC, SELinux can block access
🧮 Section 2.1.5

Memory Management

Android uses virtual memory, zRAM, and the Low Memory Killer to balance performance and memory constraints across devices.

🧮

Virtual Memory Concepts

  • Virtual address space — Each process gets its own
  • Page tables — Map virtual to physical RAM
  • Swapping to zRAM — Compressed swap in RAM
  • Page faults — Triggered accessing swapped pages
  • mmap() — Memory-mapped files for efficient I/O
📈

OOM Killer / LMK

  • oom_score_adj — Higher = more likely killed
  • Foreground apps — Score 0 (protected)
  • Cached processes — Score 900+ (killed first)
  • lmkd — Android-specific daemon
💀 Simulation: Low Memory Killer Decision Process
1
Memory Pressure Rising: Available RAM drops below threshold. LMK monitors via /proc/pressure/memory
2
LMK Threshold Reached: Minfree levels exceeded. LMK evaluates all oom_score_adj values.
3
Target Selection: Cached process (score=950) selected first, then empty, then services.
4
SIGKILL Sent: No chance to save state. Memory immediately freed.
5
Continue or Stop: If still under pressure, LMK continues killing by score order.
6
App Restart: START_STICKY services may be restarted automatically.
# Check memory status
cat /proc/meminfo          # detailed memory info
free -m                     # memory in MB

# Check process memory
dumpsys meminfo com.app     # Android memory breakdown
cat /proc/PID/smaps        # detailed memory mappings

# Check LMK settings
cat /sys/module/lowmemorykiller/parameters/minfree
cat /sys/module/lowmemorykiller/parameters/adj
🎯 Memory Forensics Tip

/proc/PID/smaps reveals sensitive data in memory including encryption keys, passwords, and decrypted data. Dump process memory before killing it during forensics.

2.2.1 — /system Partition

📁

/system Directory Structure

  • /system/app/ — System apps (pre-installed)
  • /system/priv-app/ — Privileged system apps
  • /system/framework/ — Android framework JARs
  • /system/lib/ — 32-bit shared libraries
  • /system/lib64/ — 64-bit shared libraries
  • /system/bin/ — Essential binaries
  • /system/etc/ — Configuration files
🔒

Security Properties

  • Mount: Read-only (ro)
  • SELinux context: system_file
  • Integrity: dm-verity (verified boot)
  • System apps: Run with system UID (1000)
  • Modifications: Require root + remount

2.2.2 — /data Partition

The most security-sensitive area — contains all installed application data, databases, and user information.

📦

/data Key Directories

  • /data/app/ — Installed APKs (non-system)
  • /data/data/ — App internal storage (sandbox)
  • /data/dalvik-cache/ — Optimized DEX files (OAT)
  • /data/local/ — Local temporary storage
  • /data/media/ — Internal storage emulated as /sdcard
  • /data/system/ — System settings, accounts
  • /data/system/packages.xml — All packages + permissions
🔑

Critical Files for Forensics

  • packages.xml — All packages, UIDs, permissions
  • packages.list — Package names + UIDs
  • locksettings.db — Screen lock settings
  • accounts.db — Google and other accounts
  • wifi/WifiConfigStore.xml — Saved WiFi passwords
  • system_ce/ — Credential-encrypted storage
  • system_de/ — Device-encrypted storage

2.2.3 — /cache Partition

📥

/cache Contents

  • OTA update ZIPs — Downloaded system updates
  • recovery/ — Recovery mode logs
  • data/ — App-specific cache directories
  • Temporary files — App temporary data
🗑️

Wiping /cache

  • Safe to wipe: Does not affect user data
  • Recovery mode: Wipe cache partition option
  • Forensic value: Temp files, download history
  • Anti-forensics: Attackers wipe to remove traces

2.2.4 — /sdcard (External Storage)

📱

/sdcard Structure

  • /sdcard/DCIM/ — Camera photos and videos
  • /sdcard/Download/ — Downloaded files
  • /sdcard/Android/data/ — App external storage
  • /sdcard/WhatsApp/ — WhatsApp media/databases
  • /sdcard/Telegram/ — Telegram media/documents
⚠️

Security Implications

  • No root required — Any app with READ_EXTERNAL_STORAGE
  • Scoped Storage (11+) — Apps restricted to own dirs
  • Forensic goldmine: Photos, messages, databases
  • Deleted files: May be recoverable with forensic tools
⚠️ Scoped Storage (Android 10+)

Apps can only access their own /sdcard/Android/data/com.app/ without MANAGE_EXTERNAL_STORAGE permission. Apps with this permission have full access.

2.2.5 — /proc Virtual Filesystem

/proc is a virtual filesystem providing real-time process and kernel information. It exists only in memory — no disk storage.

📊

/proc Key Files

  • /proc/cpuinfo — CPU information
  • /proc/meminfo — Memory statistics
  • /proc/version — Kernel version
  • /proc/mounts — Mounted filesystems
  • /proc/net/ — Network statistics
  • /proc/PID/ — Per-process information
🔍

Per-Process (/proc/PID/)

  • cmdline — Command line arguments
  • status — Process status (name, state, UID)
  • smaps — Detailed memory mappings
  • fd/ — Open file descriptors
  • exe — Symlink to executable
  • environ — Environment variables

🔒 Section 2.3

Application Sandboxing

Android's sandboxing model ensures each app runs in isolation, preventing unauthorized access to other apps' data and system resources.

2.3.1 — UID Separation

Every Android app is assigned a unique Linux UID at install time, used to isolate app data and restrict system resource access.

👤

How UID Assignment Works

  • Install time: PackageManager assigns UID from packages.xml
  • UID range: Apps start from 10000 (u0_aXXX)
  • Shared UID: Apps with same sharedUserId share UID
  • System apps: UIDs 1000-9999
  • Root: UID 0 (superuser)
🔐

UID-Based Security

  • File permissions: DAC checks UID/GID
  • Process isolation: Each UID = own process space
  • IPC: Binder checks caller UID/PID
  • SELinux: MAC adds another restriction layer
  • sharedUserId risk: Apps sharing UID read each other's data
Android UID Assignment Flow
APK Install
↓
PackageManager reads AndroidManifest.xml
↓
Check for sharedUserId
↓
Assign new UID
Reuse existing UID
↓
Create /data/data/com.app/ with UID:GID
⚡ Pentester's Perspective — UID
  • Shared UID exploitation: Apps sharing UID can read each other's data
  • UID enumeration: ps -A shows UID column, map via packages.list
  • root = UID 0: Root bypasses ALL UID restrictions

2.3.2 — Application Isolation

🧱

DAC Isolation

Each app's data directory owned by its UID with drwx------. Only the app (and root) can access it.

🛡️

SELinux MAC

Mandatory Access Control labels every file and process. Even if DAC allows, SELinux can block.

🔄

Seccomp-BPF

Filters syscalls per process. Apps cannot use dangerous syscalls (mount, reboot) even with root.

🧱 Simulation: Sandbox Isolation in Action
1
App A installs: Gets UID 10150. Data at /data/data/com.appA/ with drwx------
2
App B installs: Gets UID 10151. Data at /data/data/com.appB/ with drwx------
3
App B reads App A: open("/data/data/com.appA/db") → DENIED! UID mismatch
4
Even with root: SELinux (enforcing) blocks based on security contexts
5
Bypass attempt: Content Provider must be explicitly exported + permission granted
6
Defense in Depth: DAC + SELinux + Seccomp = Multiple isolation layers

ADB Fundamentals

Android Debug Bridge is the essential tool for communicating with Android devices — shell access, file transfer, app management, and debugging.

2.4.1 — What is ADB?

🔧

ADB Architecture

  • Client: Runs on computer (adb command)
  • Server: Background daemon (port 5037)
  • Daemon (adbd): Runs on Android device
  • Transport: USB, WiFi (TCP/IP), or wireless
📡

Connection Modes

  • USB: Direct cable connection
  • WiFi: Same network, TCP/IP (port 5555)
  • Wireless: Android 11+ built-in
  • Default ports: Server 5037, device 5555
ADB Communication Architecture
ADB Client (your PC)
↓ TCP:5037
ADB Server (daemon on PC)
↓ USB / TCP:5555
adbd (daemon on device)
↓
Android OS / Shell / Apps

Lab 9

Process Memory Dump

Find PID: adb shell ps -A | grep com.target.app
Maps: adb shell cat /proc/PID/maps
Open files: adb shell ls -la /proc/PID/fd/
Environment: adb shell cat /proc/PID/environ | tr '\0' '\n'
💡 Requires root. Look for keys, tokens, credentials in memory.
Lab 10

SELinux Investigation

Status: adb shell getenforce
File contexts: adb shell ls -Z /system/bin/
Process contexts: adb shell ps -eZ | grep target
App contexts: adb shell ls -Z /data/data/com.target.app/
💡 Permissive domains allow all actions but log denials — potential bypasses.
Lab 11

SQLite Database Extraction

List: adb shell su -c "ls /data/data/com.target.app/databases/"
Copy: adb shell su -c "cp .../app.db /sdcard/"
Pull: adb pull /sdcard/app.db ./
Analyze: sqlite3 app.db ".dump"
Query: sqlite3 app.db "SELECT * FROM users;"
💡 Also check .db-wal files for recent uncommitted data.
Lab 12

Full Device Forensic Acquisition

Check root: adb shell su -c "id"
Pull /data/system: adb shell su -c "tar czf /sdcard/sys.tar.gz /data/system/"
Pull app data: for app in /data/data/*; do tar czf /sdcard/$(basename $app).tar.gz $app; done
Pull /sdcard: adb pull /sdcard/ ./sdcard_backup/
Dump packages: adb shell su -c "cat /data/system/packages.xml"
💡 Document all steps with timestamps for forensic chain of custody.

📚 Key Terms

Module 2 Flashcards

Click any card to flip and reveal the definition.

SELinux

Click to reveal

Security-Enhanced Linux — Mandatory access control. Runs in enforcing mode on Android, confining processes beyond traditional DAC.

Binder IPC

Click to reveal

Android's custom IPC mechanism. Replaces traditional Linux IPC. Thread-safe, synchronous with built-in UID/PID verification.

OOM Killer / LMK

Click to reveal

Low Memory Killer Daemon. Android-specific OOM killer using oom_score_adj. Cached processes die first under memory pressure.

zRAM

Click to reveal

Compressed RAM swap. Creates compressed swap in memory instead of slow flash storage. ~2x effective memory on Android.

setuid (suid)

Click to reveal

Set User ID bit. Binary runs with file owner privileges (often root). Common privilege escalation vector.

Seccomp-BPF

Click to reveal

Secure Computing mode. Filters syscalls per process. Prevents dangerous syscalls (mount, reboot) even with root.

UID Isolation

Click to reveal

Unique Linux UID per app. File permissions ensure only the owning UID can access the app's data directory.

Content Provider

Click to reveal

Structured data sharing. Apps must explicitly export providers. Unintentionally exported providers are common vulnerabilities.

packages.xml

Click to reveal

Master package database. /data/system/packages.xml lists every package, UID, path, and granted permission. Forensic goldmine.

ADB

Click to reveal

Android Debug Bridge. Tool for shell access, file transfer, app management, and debugging. Essential for security testing.

dm-verity

Click to reveal

Verified Boot. Block-level verification of /system at boot. Detects tampering. Part of Android's Verified Boot chain.

Project Treble

Click to reveal

OEM/vendor separation. Splits /system from /vendor. Independent updates via HIDL hardware abstraction.
